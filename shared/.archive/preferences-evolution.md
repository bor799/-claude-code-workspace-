### 知识库目录分类修复

**来源**: 职业发展目录整理会话
**洞察**: 发现并修复了 `manage.sh` 脚本中 `infer_category()` 函数重复定义导致分类错误的根本问题
**细节**:
- **问题**: `infer_category()` 有两个定义（第157行和第621行），后者覆盖前者
- **症状**: 知识库类型目录的分类使用了基于关键词的推断，而非目录结构推断
- **根本原因**: 第621行的 `infer_category()` 根据 `CATEGORIZE_MODE` 选择逻辑，默认值 `auto` 导致使用 `infer_category_from_content()`
- **修复方案**:
  1. 删除第157-170行被覆盖的旧函数
  2. 修改第621-630行的函数，让 `knowledge_base` 类型自动使用目录结构推断

**修复代码**:
```bash
infer_category() {
    local file=$1
    local mode="${CATEGORIZE_MODE:-auto}"

    # 知识库模式：优先使用目录结构
    if [ "$DIR_TYPE" = "knowledge_base" ]; then
        infer_category_from_structure "$file"
        return
    fi

    # 信息源模式：根据 CATEGORIZE_MODE 选择
    if [ "$mode" = "structure" ]; then
        infer_category_from_structure "$file"
    else
        infer_category_from_content "$file"
    fi
}
```

**行动**:
1. 修复 `manage.sh` 脚本
2. 使用 sed 批量修复已存在的分类错误
3. 重新生成看板验证结果

**影响**:
- `兴趣领域` 和 `职业发展` 两个知识库目录现在可以正确使用目录结构推断分类
- 未来添加新的知识库目录将自动受益于此修复
- 避免了每次都需要手动修复分类的问题

**置信度**: 0.90 (↑ 0.08, 根因问题已彻底修复)

- **红旗问题**: 过度自主实现 (-12分) - 在用户未明确要求的情况下创建复杂系统
- **技术问题**:
  - tmux socket 权限问题: `Operation not permitted`（macOS 沙盒限制）
  - Python SSL 模块损坏: Homebrew Python 缺少 OpenSSL 库
  - screen session 无法保持运行
- **解决方案**:
  - 使用系统 Python (`/usr/bin/python3`) 代替 Homebrew Python
  - 直接运行 Python 脚本 + 后台运行，不使用 tmux/screen
  - 简单的轮询模式实现 Telegram 消息接收

**关键学习**:
1. **模糊需求需澄清**: 当用户提出"Agent军队"等概念时，应先询问：
   - "你想要多个独立的Bot（不同Token），还是同一个Bot的多个实例？"
   - "这些Agent需要不同的角色分工吗？"
2. **简单方案优先**: 除非用户明确要求复杂度，否则从最简单的方案开始
3. **技术备用方案**: macOS 环境下 tmux 可能不可用，应准备备用方案

1. 创建简单的 `bridge.py` 脚本（单文件，无依赖）
