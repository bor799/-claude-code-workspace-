# FractalFlow 分形文档协议

## 核心思想

每个子项目是一个"分形单元"。层级之间通过强制同步协议保持一致。
在任意子目录下，Claude 都能通过该目录的文档重建局部世界观。

## 三级文档体系

### 层级一：根目录宪法（CLAUDE.md）

位置：`~/CLAUDE.md`
职责：全局规则、专家路由表、工具优先级
原则：极简，只写全局约束，不写任何子项目细节

### 层级二：文件夹地图（.folder.md）

位置：`<子项目>/.folder.md`
职责：子项目的"局部宪法"，三行极简原则
原则：让 Claude 进入目录后第一眼就知道"这是什么、怎么运作、有什么约束"

#### 标准模板

```markdown
# <项目名>

> **地位**：<一句话定义这个项目在全局中的位置>
> **逻辑**：<输入什么 → 处理什么 → 输出什么>
> **约束**：<铁律，违反即出错>

## 成员清单

| 文件/目录 | 角色 | 说明 |
|-----------|------|------|
| `SKILL.md` | 入口 | 触发词、技能定义 |
| `agents/` | 工作者 | 专业 Agent 集合 |
| `scripts/` | 执行器 | 脚本工具 |
| `references/` | 知识库 | 参考模板 |

## 依赖关系

| 方向 | 项目 | 说明 |
|------|------|------|
| 依赖 | `../xxx/` | 需要什么 |
| 被依赖 | `../yyy/` | 谁需要我 |

## ⚠️ 触发器

一旦本文件夹增删文件或架构逻辑调整，请立即重写此文档。
```

### 层级三：文件头注释

位置：`SKILL.md` 的 frontmatter、脚本文件的开头注释
职责：文件的 INPUT/OUTPUT/POSITION 自描述

#### SKILL.md frontmatter 标准

```yaml
---
name: <技能名>
version: <版本>
description: <一句话描述>
trigger:
  - <触发词列表>
allowed-tools:
  - <工具列表>
---
```

#### 脚本文件头标准（如有脚本）

```
# [INPUT]: <接收什么>
# [OUTPUT]: <产出什么>
# [POS]: <在项目中的位置和角色>
```

## 同步协议（强制）

### 规则 1：原子更新

任何功能、架构、写法更新，必须在代码修改完成后，
**立即**同步更新对应层级的文档。

### 规则 2：递归触发

```
文件变更 → 更新文件头（层级三）
         → 更新所属文件夹 .folder.md（层级二）
         → 若影响全局，更新 CLAUDE.md（层级一）
```

### 规则 3：分形自治

确保系统在任何一个子目录下，Claude 都能通过：
1. `.folder.md` 了解项目全貌
2. `SKILL.md` 了解触发和使用方式
3. 文件头注释了解单个文件的职责

而不需要回溯到根目录的 CLAUDE.md。

## 操作检查清单

创建或修改子项目时，按此清单检查：

- [ ] `.folder.md` 的三行定义（地位/逻辑/约束）是否准确
- [ ] 成员清单是否与实际文件一致
- [ ] 依赖关系是否正确
- [ ] `SKILL.md` frontmatter 是否完整
- [ ] 脚本文件头是否有 INPUT/OUTPUT/POS
- [ ] 变更是否触发了上级文档更新

## 适用于本项目的情况

本工作区是 Claude Code 技能/插件系统，不是传统代码项目。
因此：

- **层级二** (.folder.md) 适用于每个子项目目录
- **层级三** 文件头主要适用于 `SKILL.md` 和 `scripts/` 下的脚本
- **不需要**给每个 .md 参考文件加头注释（references/ 目录豁免）

## 新子项目初始化流程

```
1. 创建目录 <项目名>/
2. 复制 .folder.md 模板，填写三行定义
3. 创建 SKILL.md，填写标准 frontmatter
4. 创建必要的子目录（agents/, scripts/, references/）
5. 更新父级文档（如有）
```
